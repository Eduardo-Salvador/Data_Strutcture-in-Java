**Time vs. Space:**

- An algorithm can be fast but consume a lot of memory, or consume little memory but be slow. Big-O measures both: time (execution) and space (memory used).
- You can cache results to avoid calculations; this uses space but saves time.
- Always consider the worst-case scenario, where `n` is the number of operations needed.

**Complexities:**

- O(1) ‚Üí **Constant time** (Ex: Accessing an item in an array)
- O(logN) ‚Üí **Logarithmic** (Ex: Binary search)
- O(n) ‚Üí **Linear** (Ex: Traversing a list)
- O(n log N) ‚Üí **Good for sorting algorithms**.
- O(n2) ‚Üí **Bad:** Algorithms with many nested loops.
- **Temporal complexity** ‚Üí execution time
- **Spatial complexity** ‚Üí memory usage
- **Time** = how many times you *process* data
- **Space** = how much data you *store at the same time*

**It's not always possible to have the perfect algorithm, but it can be improved.**

- Avoid unnecessary loops.
- Use more efficient data structures (HashMap instead of List). - Divide and Conquer (MegeSort).
- Use cache (memorization).
